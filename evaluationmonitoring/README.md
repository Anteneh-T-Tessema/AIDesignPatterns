# Agentic Design Pattern: Evaluation & Monitoring

This directory implements the **Evaluation & Monitoring** agentic design pattern (inspired by Chapter 19 of *Agentic Design Patterns*). 

Because AI agents operate probabilistically and execute multi-step plans using tools, standard software tests (binary assertions) are insufficient for verifying reliability. Robust production systems must track and audit **semantic accuracy**, **execution latency**, **cost/token usage**, **step trajectories (tool call order)**, and **qualitative response quality** over time.

---

## Pattern Overview

The Evaluation and Monitoring pattern establishes a structured observation layer around active agents:

```
                  ┌──────────────────────────────┐
                  │      User Travel Query       │
                  └──────────────┬───────────────┘
                                 │
                                 ▼
                     ┌───────────────────────┐
                     │ TravelAssistantAgent  │ ◄───┐
                     └───────────┬───────────┘     │
                                 │                 │
                                 ▼ (Tool Traces,   │ A/B Iteration
                            Latency, Tokens, Text) │ Feedback
                                 │                 │ Loop
                                 ▼                 │
                     ┌───────────────────────┐     │
                     │    Evaluator Suite    ├─────┘
                     │                       │
                     │  - Accuracy Metrics   │
                     │  - Latency / Cost     │
                     │  - Trajectory Auditor │
                     │  - LLM-as-a-Judge     │
                     └───────────────────────┘
```

## Core Metrics

### 1. Syntactic & Semantic Accuracy
- **Exact Match**: Direct character-level check (ideal for key structural words or parameters).
- **Levenshtein Distance**: Measures editing differences (character substitutions/insertions/deletions) to evaluate proximity to reference text.
- **Jaccard Similarity**: Evaluates word overlap (intersection divided by union), catching paraphrasing and scrambled orders.

### 2. Trajectory Auditing (Tool Verification)
Tracks the exact array of tool invocations generated by the agent against a "ground truth" list (e.g., `["lookupFlights", "checkWeather", "bookHotel", "applyDiscount"]`):
- **Exact Match**: The agent called precisely the expected tools in the exact order.
- **In-Order Match**: The agent called all expected tools sequentially, but may have inserted retries or extra lookup steps in between.
- **Any-Order Match**: The agent called all necessary tools, but the sequence order was incorrect.
- **Precision / Recall / F1 Score**: Quantifies correct tools invoked versus redundant/missing steps.

### 3. Resource & Performance Monitoring
- **Latency Tracker**: Computes cumulative runtimes as well as step-by-step tool processing durations.
- **Token Monitor**: Tracks prompt and completion tokens to calculate cost metrics and detect overhead changes.

### 4. LLM-as-a-Judge
Leverages a grading rubric prompt sent to an LLM evaluator to score subjective qualities (e.g. *Accuracy* and *Helpfulness*) on a 1-5 scale. Provides a rule-based fallback when offline.

## File Structure

- `package.json` — Configuration, scripts, and dependencies.
- `evaluator.js` — Accuracy, latency, token, trajectory, and LLM-as-a-Judge algorithms.
- `agent.js` — `TravelAssistantAgent` supporting optimized Version A and inefficient/redundant Version B.
- `index.js` — Interactive CLI application detailing four core scenarios:
  1. Optimized Execution (Version A validation)
  2. Trajectory Drift Detection (Version B validation)
  3. Version Comparison (A/B testing analysis)
  4. LLM-as-a-Judge Qualities Rubric
- `README.md` — This documentation file.

---

## Usage

1. Install dependencies from this directory:
   ```bash
   npm install
   ```

2. Run the interactive CLI:
   ```bash
   npm start
   ```

3. If you have **Ollama** running locally with the `llama3.2` model, the CLI will automatically attempt to run live evaluations. Otherwise, it will run using deterministic mock pipelines to ensure accuracy metrics can be observed instantly.
